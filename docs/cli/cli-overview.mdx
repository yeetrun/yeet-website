---
title: How Commands Work
description: How yeet chooses a host, chooses a service, and separates its flags from payload args.
---

Yeet runs on your workstation. Catch runs on remote hosts. Most yeet commands
send requests to the selected catch host.

The command model has three recurring questions: which host, which service, and
which arguments belong to yeet versus the payload. When a command looks weird,
one of those boundaries is usually the culprit.

## Pick the host

Setup uses an SSH machine host:

```bash
yeet init root@<machine-host>
```

Normal commands use the catch host:

```bash
CATCH_HOST=<catch-host> yeet status
yeet --host=<catch-host> status
yeet status@<catch-host>
```

Run new-service deploy commands from a
[Service Workspace](/docs/getting-started/service-workspace) so the generated
`yeet.toml` lands in the directory you intend to use for that service set.

Target a specific host when deploying:

```bash
yeet run <svc>@<catch-host> ./compose.yml
```

If `yeet.toml` contains a host list, `yeet status` with no service queries
those hosts. Project state wins over guessing, which is the correct bias for a
tool that can restart things.

Save a default host:

```bash
yeet prefs --host=<catch-host> --save
```

## Pick the service

Most service commands accept a service name:

```bash
yeet status <svc>
yeet status <svc-a> <svc-b>
yeet info
yeet info <svc>
yeet logs -f <svc>
```

New service names must use lowercase letters, numbers, and dashes, start with a
letter, and end with a letter or number.

Use `--service=<svc>` only when a command needs an explicit override:

```bash
yeet --service=<svc> logs -f
```

Some commands are host-level and do not target a service, such as `prefs`,
`list-hosts`, `mount`, and `umount`. Not every operation needs a service name;
forcing one would just create fake precision.

## Pass args to payloads

Use `--` when arguments belong to the service payload instead of yeet:

```bash
yeet run <svc> ./bin/<svc> -- --flag value
yeet cron <svc> ./job.sh "0 9 * * *" -- --job-arg foo
```

This keeps yeet flags and application flags separate. Without the separator,
every parser in the chain has to guess where its job ends. Parsers are bad at
guessing politely.

## Get help

Use standard help for people and structured help for AI tools or scripts:

```bash
yeet --help
yeet run --help
yeet run --help-agent
```

Use [yeet Commands](/docs/cli/yeet-cli) for a grouped command index.

## TTY commands

Interactive commands such as `yeet edit`, `yeet env edit`, and some prompts
need a TTY. Run them from an interactive shell. For scripts, prefer explicit
flags such as `--yes`, `--progress=plain`, or `--progress=quiet` where the
command supports them. Automation should not depend on pretending a prompt is a
person.
