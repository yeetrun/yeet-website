Use a VM payload when a workload needs a long-lived Linux guest, its own init
system, or stronger isolation than a container.

VM payloads require a Linux catch host with KVM. If you are setting up a host
for the first time, run the container smoke test before starting here.

VMs are useful because they are real machines. They are also annoying because
they are real machines. They boot, consume memory, own disks, have networking,
and keep secrets in places containers usually do not. Use them when that is the
tradeoff you actually want.

Run VM creation commands from a
[Service Workspace](/docs/getting-started/service-workspace). The first
successful VM deploy writes `yeet.toml` in the current directory.

## Create a VM

List the official catalog:

```bash
yeet vm images catalog
```

Create an Ubuntu VM:

```bash
yeet run <vm> vm://ubuntu/26.04
yeet ssh <vm>
```

Create a NixOS VM:

```bash
yeet run <vm> vm://nixos/26.05
yeet ssh <vm>
```

`yeet run` creates the VM service. It is not the update path for an existing VM
with the same name. Use `yeet vm set` for supported changes, or remove and
recreate the VM when you want a fresh guest. A VM has state; redeploying it
like a stateless container would be a neat way to lose the plot.

Override the starting shape when needed:

```bash
yeet run <vm> vm://ubuntu/26.04 --vcpus=4 --memory=4g --disk=128g
```

## Choose VM memory behavior

`--memory` is the VM's maximum RAM. New VMs enable Firecracker ballooning by
default, so catch can reclaim guest memory under host pressure without lowering
the configured maximum. This is a pressure valve, not a promise that memory is
infinite.

Set the minimum floor when you create the VM:

```bash
yeet run <vm> vm://ubuntu/26.04 --memory=4g --memory-min=1g
```

`--memory-min` is the floor yeet will not intentionally reclaim below. Change
it later with:

```bash
yeet vm set <vm> --memory-min=2g
```

Disable ballooning when a VM should reserve its full maximum memory:

```bash
yeet vm set <vm> --balloon=off
```

Host memory policy defaults to `safe`, which admits VMs against their maximum
RAM. Opt into controlled overcommit only when you want catch to use VM memory
floors for admission. Overcommit is a capacity bet. Make it on purpose:

```bash
yeet vm memory set --policy=balanced
```

## Choose VM networking

`svc` networking is the default. It gives yeet a private management path, yeet
DNS, and outbound internet through the catch host:

```bash
yeet run <vm> vm://ubuntu/26.04 --net=svc
```

Use `lan` when the VM should request a LAN or VLAN address through DHCP. The VM
uses that network's gateway and DNS for ordinary internet access. If that
network has bad DHCP, bad DNS, or no gateway, the VM faithfully inherits the
mess:

```bash
yeet run <vm> vm://ubuntu/26.04 --net=lan
```

For VM LAN networking, catch attaches the guest TAP device to a host bridge. If
the host already has a bridge on the default LAN route, yeet uses it. On
supported Debian/Ubuntu hosts without a bridge, yeet can prepare `br0` from the
default LAN uplink after you confirm.

Interactive `yeet init` offers that bridge preparation when VM support is
available. You can also skip it during init; the first `yeet run <vm> ...
--net=lan` prompts before creating VM files or database state. If the host
briefly drops the connection while applying the bridge, wait for it to come
back and rerun the same `yeet run` command. Bridge changes are real host
network changes, not decorative YAML.

Use `svc,lan` when the VM needs host LAN access and should still keep a
catch-managed fallback path:

```bash
yeet run <vm> vm://ubuntu/26.04 --net=svc,lan
```

With `svc,lan`, yeet uses the LAN IP when your workstation can reach it and
falls back to the catch proxy when it cannot. Force the proxy path with:

```bash
yeet ssh --force-proxy <vm>
```

VM network modes do not include `ts`. If the guest needs its own Tailscale
identity, install and configure Tailscale inside the guest. At that point the
guest is the actor joining the tailnet, which is the right boundary.

See [Networking](/docs/concepts/networking) and [DNS](/docs/concepts/dns) for
service-network names and LAN behavior.

## Connect and copy files

Open an interactive shell:

```bash
yeet ssh <vm>
```

Run one command:

```bash
yeet ssh <vm> -- uname -a
```

Stream the serial console for boot diagnostics:

```bash
yeet vm console <vm>
```

Detach from the console by pressing Enter, then typing `~.`. The VM keeps
running. The console is a window into the guest, not a lifecycle command.

Copy files with VM endpoints:

```bash
yeet copy ./app <vm>:~/app
yeet copy ./configs/*.yml <vm>:~/configs/
yeet copy <vm>:"/var/log/*.log" ./logs/
yeet copy --force-proxy ./configs/ <vm>:~/configs/
```

Ubuntu VMs default to the `ubuntu` user. NixOS VMs default to the `nixos` user.

## Change an existing VM

Stop the VM before changing CPU, memory, disk size, or networking:

```bash
yeet stop <vm>
yeet vm set <vm> --vcpus=6 --memory=6g
yeet vm set <vm> --disk=128g
yeet vm set <vm> --net=svc,lan
yeet start <vm>
```

Disk changes are grow-only. Shrinking disks and live resize are not supported.
Disks are where optimism goes to meet filesystems.

For VLAN-tagged LAN networking:

```bash
yeet stop <vm>
yeet vm set <vm> --net=svc,lan --macvlan-parent=<iface> --macvlan-vlan=<id>
yeet start <vm>
```

## Host requirements

VM payloads require:

- x86_64/amd64 Linux.
- KVM at `/dev/kvm`.
- TUN/TAP at `/dev/net/tun`.
- `qemu-img`, `zstd`, `e2fsck`, `resize2fs`, `mount`, `umount`, and `ip`.

On Debian/Ubuntu hosts with KVM and TUN/TAP available, interactive `yeet init`
can prompt to install missing VM tools and prepare VM LAN bridge support. For
unattended VM tool setup, pre-answer that prompt with:

```bash
yeet init --install-vm-tools root@<machine-host>
```

Automatic VM LAN bridge preparation currently supports Debian/Ubuntu
netplan-networkd hosts with a physical default-route LAN uplink. Hosts that
already route through a bridge reuse that bridge without rewriting the host
network.

Many VPS providers do not expose nested virtualization. Use containers,
binaries, scripts, and cron jobs on those hosts, and run VMs on a KVM-capable
machine. This is not a yeet limitation so much as physics with a provider
invoice.

## Image cache

Catch caches official VM images per host. A missing image downloads
automatically on the first VM create.

```bash
yeet vm images catalog
yeet vm images
yeet vm images update
yeet vm images update vm://nixos/26.05
yeet vm images prune --dry-run
```

`yeet vm images update` refreshes official image families for future VM
creates. It does not rewrite existing VM disks. Existing VMs are stateful
guests, not templates waiting to be surprised.

When creating a VM with a stale cached image, interactive runs prompt by
default. For non-interactive runs, choose explicitly:

```bash
yeet run <vm> vm://ubuntu/26.04 --image-policy=update
yeet run <vm> vm://ubuntu/26.04 --image-policy=cached
```

Preview cleanup before removing cached image data:

```bash
yeet vm images prune --dry-run
```

## ZFS-backed VM disks

ZFS is optional. Use it when you want storage-level recovery points or faster
repeated VM creates on the same pool:

```bash
yeet run <vm> vm://ubuntu/26.04 --service-root=tank/vms/<vm> --zfs
```

For ZFS-backed VMs, yeet can clone a shared image base on the same pool instead
of writing the root filesystem again. Raw VM disks work without ZFS. Faster
clone paths are useful; optional dependencies are still optional.

See [ZFS](/docs/concepts/zfs) for dataset, snapshot, and cleanup behavior.

## VM snapshots and restore

Create a manual recovery point:

```bash
yeet snapshots create <vm>
yeet snapshots create <vm> --comment "before package upgrade"
```

List and inspect recovery points:

```bash
yeet snapshots list <vm>
yeet snapshots inspect <vm> <snapshot>
```

Clone first when you want to inspect a recovered disk without changing the
original VM. It is the recovery version of "measure twice, cut once," except
with less woodworking and more panic avoidance:

```bash
yeet snapshots clone <vm> <snapshot> <new-vm>
yeet info <new-vm>
```

Restore the original VM disk in place:

```bash
yeet snapshots restore <vm> <snapshot> --mode=disk --stop --yes
```

`--mode=disk` is the default. Add `--start` when the VM should start again
after restore.

Full VM checkpoints are available for advanced recovery:

```bash
yeet snapshots create <vm> --full --comment "checkpoint before risky change"
yeet snapshots restore <vm> <snapshot> --mode=full --stop --yes
```

Use disk restore unless you specifically need to resume from a Firecracker
checkpoint. Full checkpoints are powerful and specific; do not reach for them
just because the word "full" sounds safer.

## Import a local VM image

Use local images when you have built your own root filesystem bundle and want
catch to cache it by name:

```bash
yeet vm images import lab/ubuntu ./dist/my-vm
yeet run devbox vm://lab/ubuntu
```

The import directory must contain `rootfs.ext4` or `rootfs.ext4.zst`. If it
also contains a yeet VM `manifest.json`, yeet uses the guest settings from that
manifest. Re-importing the same image name affects new VMs only; existing VMs
keep the image version from their original create. Image names are cache keys
for future guests, not remote-control handles for existing ones.

Remove an imported image:

```bash
yeet vm images rm lab/ubuntu --yes
```

## Cleanup

Uninstall the VM service but keep its disk:

```bash
yeet rm <vm>
```

Delete the guest disk and local config entry too:

```bash
yeet rm --clean <vm>
```

For ZFS-backed VMs, cleanup removes the VM clone and service data. It does not
remove shared image bases that future VMs can reuse. Shared bases are cache
state, not the guest you just removed.
