VM payloads create long-lived Ubuntu guests managed by yeet. They are useful
when a workload needs a full guest OS or stronger isolation than a container.

## Create a VM

`svc` networking is the default for VM payloads:

```bash
yeet run <vm> vm://ubuntu/26.04
yeet ssh <vm>
```

Use `lan` when the VM should request an address on your LAN through DHCP:

```bash
yeet run <vm> vm://ubuntu/26.04 --net=lan
```

Override the default shape when needed:

```bash
yeet run <vm> vm://ubuntu/26.04 --cpus=4 --memory=4g --disk=128g
```

## Change an existing VM

Stop a VM before changing CPU, memory, disk size, or networking:

```bash
yeet stop <vm>
yeet service set <vm> --cpus=6 --memory=6g
yeet service set <vm> --disk=128g
yeet service set <vm> --net=lan
yeet start <vm>
```

CPU and memory changes update the Firecracker config used on the next start.
Disk changes are grow-only: yeet expands the raw disk file or ZFS ZVOL, runs an
ext4 check, and grows the root filesystem. Shrinking disks and live resize are
not supported.

Network changes replace the VM's network interfaces while the VM is stopped.
Use `--net=svc`, `--net=lan`, or a comma-separated list such as
`--net=svc,lan`. For LAN networking, `--macvlan-parent`, `--macvlan-vlan`, and
`--macvlan-mac` follow the same rules as VM creation.

For VLAN-tagged LAN networking, pass `--net=lan --macvlan-vlan=<id>`. yeet uses
standard Linux VLAN and bridge devices: it creates its own VLAN path when it can,
or uses an existing Linux bridge for that VLAN if the host already has one.

## Host requirements

VM payloads require a Linux catch host with KVM available. During `yeet init`,
catch checks for KVM, TUN/TAP, and required VM tooling such as `qemu-img`,
`zstd`, `e2fsck`, `resize2fs`, `mount`, `umount`, and `ip`. On Debian/Ubuntu
hosts with the required VM devices available, interactive installs can offer to
install missing VM packages.

ZFS is optional unless you create VMs with `--zfs`. See
[ZFS](/docs/concepts/zfs) for the shared dataset and snapshot model.

## ZFS-backed VM disks

```bash
yeet run <vm> vm://ubuntu/26.04 --service-root=tank/vms/<vm> --zfs
```

For ZFS-backed VMs, the first VM created on a pool for an image version prepares
a shared image base on that pool. Later VMs on the same pool and image version
clone that shared base instead of writing the root filesystem again.

Use ZFS-backed VMs when you want fast repeated VM creates on the same pool or
storage-level VM disk management. Raw VM disks still work without ZFS.

Growing a ZFS-backed VM disk grows the VM ZVOL and then the guest ext4
filesystem. `yeet service set <vm> --disk=<size>` still requires the VM to be
stopped and still refuses shrink requests.

## Image cache

VM image bundles are cached per catch host.

```bash
yeet vm images
yeet vm images update
yeet vm images prune --dry-run
```

A missing image is downloaded automatically on the first VM create. Refreshing
the cache updates the image used for future VM creates, does not rewrite
existing VM disks, and automatically removes old unreferenced image versions
when it is safe. When creating a VM with a stale cached image, interactive runs
prompt by default; non-interactive runs require `--image-policy=update` or
`--image-policy=cached`.

`yeet vm images prune --dry-run` shows what cleanup would remove. Run
`yeet vm images prune` to review the same plan and confirm removal, or add
`--yes` to skip the prompt.

## Local VM images

Import a local rootfs bundle onto a catch host, then run it by name:

```bash
yeet vm images import lab/ubuntu ./dist/my-vm
yeet run devbox vm://lab/ubuntu
yeet vm images rm lab/ubuntu --yes
```

A local bundle is a directory containing `rootfs.ext4` or `rootfs.ext4.zst`.
If the directory also contains a yeet VM `manifest.json`, yeet preserves the
guest boot capability metadata from that manifest. Rootfs-only bundles use
yeet's managed kernel and Firecracker from the current Ubuntu VM image. To test
a local kernel, include `vmlinux` in the bundle and import with
`--allow-local-kernel`:

```bash
yeet vm images import kernel/test ./dist/my-vm --allow-local-kernel
```

Imported image names are catch-host global. Re-importing the same name updates
the ref for new VMs; existing VMs keep the exact imported content version they
were created with.

## Boot and readiness

When `yeet run` starts a VM, it waits for the guest to report SSH readiness and
an IPv4 address before printing the next `yeet ssh` command.

```bash
yeet ssh <vm>
yeet vm console <vm>
```

Use `yeet ssh <vm>` for an interactive guest shell. Use `yeet vm console <vm>`
for serial output and boot diagnostics. For yeet-managed VM aliases in
`~/.yeet/known_hosts`, `yeet ssh <vm>` repairs a stale host key and retries
once after a VM is recreated. Yeet does not edit normal `~/.ssh/known_hosts`
entries.

The default `ubuntu` user has passwordless sudo. Metadata injection also writes
yeet-managed `.profile` and `.bashrc` blocks for that user, including
`~/.local/bin` on `PATH`, `XDG_RUNTIME_DIR` when systemd creates it, and
color-friendly Ubuntu shell behavior. User content outside the managed blocks is
preserved.

## Kernel and init

The fast Ubuntu VM image uses a yeet-managed Firecracker kernel and init shim.
Guest apt upgrades do not manage the boot kernel, bootloader, initrd, or init
shim. Publish a new yeet VM image bundle to update that boot path.

The official image includes built-in TUN, IPv6, nftables, conntrack, conntrack
marks, and NAT/masquerade support, plus `iptables` and `nftables` userspace
tools, so guest-installed router and VPN services can manage their own firewall
and routing state. Tailscale is not preinstalled.

The fast VM image does not support snap packages. Use container payloads for
workloads that need snap-like packaging behavior.

## Cleanup

```bash
yeet rm <vm>
yeet rm --clean-data <vm>
```

Use `--clean-data` when you want to delete the guest disk too. For ZFS-backed
VMs, cleanup removes the VM clone and service data, not the shared image base.
