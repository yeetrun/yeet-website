## June 22, 2026

### v0.8.9

- VM startup now waits for the guest agent and SSH readiness signal before reporting the VM as ready, making first boot and rebuild flows less likely to race guest setup.

## June 21, 2026

### v0.8.8

- New NixOS VMs now seed `/etc/nixos/system.nix` with the VM service name, so the guest hostname matches `yeet run <name>` and stays aligned after `nixos-rebuild`.

### v0.8.7

- VM guest kernel upgrades now work through the normal guest flow: install the new `yeet-vm-kernel` package or rebuild NixOS, reboot the VM, and catch syncs the selected kernel before Firecracker starts it again.
- VM image metadata now preserves kernel provenance and compares image revisions more consistently, so catalog image updates and kernel-versioned image tags select the intended kernel build.
- `yeet ip` and `yeet info` now keep user-connectable endpoints separate from runtime-only network addresses, making service networking output easier to act on.

## June 20, 2026

### v0.8.6

- VM deploys now show the same compact progress style as other `yeet run` workflows, with timed stages, clearer waiting states, and more useful next-step commands when provisioning finishes or fails.
- Interactive VM image prompts now pause the progress display cleanly, so image downloads and setup confirmations are easier to read and answer.

### v0.8.5

- VMs with both service and LAN networking now prefer the LAN route for normal traffic and use a reachable LAN SSH path first, while the service network remains available for yeet management.
- Service-network workloads keep public internet access while yeet blocks direct private-network and host traffic by default.
- Service-network repair now preserves published ports and joined port mappings more reliably after catch or Docker network reconciliation.

### v0.8.4

- `yeet copy` now accepts multiple sources, so shell-expanded globs and VM remote glob downloads work with the same last-argument-is-destination convention as scp and rsync.
- `yeet run --web` now filters payload choices as you type and validates required Tailscale tags before deploy, making the guided flow faster to drive from the keyboard and clearer before submit.
- VM networking and SSH are more predictable: VMs get a service-network gateway path, remote SSH commands preserve their argument boundaries, and `yeet info` reports missing services more clearly.

## June 19, 2026

### v0.8.3

- Compose deploys no longer require a local Docker daemon when there are no local images to push, so `yeet run --web` can deploy from machines where the local Docker runtime is stopped.
- Failed first-time web deploys now clean up staged placeholders, avoiding misleading "service already exists" errors on the next attempt.
- ZFS-backed Compose deploys now keep uploaded env files under the selected service root, and LAN plus Tailscale services no longer fail startup because of a missing namespace resolver file.

### v0.8.2

- VM service networking now routes traffic through the service bridge correctly, avoiding hangs when VMs reach yeet service-network apps by name or service IP.
- Tailscale-exposed services now repair their sidecar networking after catch recreates a service network namespace, so services keep responding after netns repair, catch restarts, or host reboots.

### v0.8.1

- VM service networking is repaired automatically after catch restarts, host reboots, and VM setting changes, and `yeet ssh` now reaches VMs with both service and LAN networking through the VM's LAN address via catch.
- VMs that shut themselves down from inside the guest are now detected as stopped more reliably, including Firecracker guests that report a halted state after `poweroff`.
- `yeet vm images catalog` now shows the fields users need for choosing a VM payload without exposing empty kernel-policy details for managed catalog images.

## June 17, 2026

### v0.8.0

- Added `yeet snapshots` recovery commands for ZFS-backed services and VMs, including manual recovery points, list and inspect views, clone-first recovery, in-place restore, protection from pruning, and full VM checkpoints.
- `yeet run --web` now discovers ZFS roots from the selected host, fills predictable VM sizing defaults, and keeps VM network choices valid while still allowing service-network, LAN, or combined VM networking.
- Service-network deployments now get yeet-managed DNS, and official VM images resolve through the published catalog so hosts can pick up catalog updates without a yeet binary release.

## June 12, 2026

### v0.7.2

- `yeet run --web` is now a workload-first new-service builder for Compose apps, VM catalog images, Dockerfiles, container images, binaries/scripts, and scheduled jobs, with a command preview that matches the deploy path.
- The web builder now keeps common network, storage, and ZFS choices visible while showing workload-specific fields, including safer VM defaults and cleaner image-reference entry.
- `yeet logs <service>` now streams through a non-TTY path and handles client disconnects cleanly, avoiding inconsistent output and broken-pipe noise.

### v0.7.1

- VM setup now fails earlier for malformed network modes, unsupported VM networking, invalid VLAN values, and macvlan settings that are missing LAN networking.
- Failed VM provisioning cleans up new service roots more safely while preserving existing service-root data, including existing ZFS datasets.
- `yeet upgrade` handles protected install directories more reliably, and VM image manifests now validate guest runtime metadata before use.

## June 10, 2026

### v0.7.0

- `yeet upgrade` can now replace a protected local yeet binary through `sudo`, fixing self-upgrades from root-owned install directories.
- VM creates and `yeet vm set` now reject unsupported VM network modes and LAN-only macvlan settings before provisioning or changing a service.
- Imported local VM images now keep guest metadata correct: default users appear in `yeet vm images catalog`, metadata changes get distinct image IDs, and older imported refs still work.

## June 9, 2026

### v0.6.16

- `yeet copy` now syncs VM endpoints into the guest filesystem with rsync over the same direct or proxied SSH path as `yeet ssh`.
- Official Ubuntu and NixOS VM images include rsync for VM guest file copy.

### v0.6.15

- Moved VM resource and networking changes to `yeet vm set`, keeping `yeet service set` focused on service roots, published ports, and snapshots.

## June 8, 2026

### v0.6.14

- Added `yeet vm images catalog` so catch hosts can list runnable VM payloads, including official and imported local images, before any image is cached.

### v0.6.13

- Release downloads now keep retrying transient GitHub edge failures long enough for freshly published assets to become available.

### v0.6.12

- Release downloads used by `yeet init --from-github` and `yeet upgrade` now retry transient GitHub fetch failures.

### v0.6.11

- VM image downloads now retry transient GitHub fetch failures and identify yeet as the downloader, making new image rollouts more reliable.
- VM SSH host-key repair now clears stale backup files before removing regenerated guest keys from yeet's known-hosts file.

### v0.6.10

- Fixed SSH access for fresh `vm://nixos/26.05` VMs by writing per-user OpenSSH authorized-key metadata that NixOS consumes through normal key-file configuration.

### v0.6.9

- Improved `vm://nixos/26.05` SSH metadata permissions so yeet-written key files are readable by the guest OpenSSH configuration.

### v0.6.8

- Fixed VM guest init so non-Ubuntu images can read their kernel command line before starting systemd.

### v0.6.7

- Fixed official VM image channels so Ubuntu and NixOS latest manifests resolve independently.

## June 7, 2026

### v0.6.6

- Added `vm://nixos/26.05` as an official Firecracker VM image alongside Ubuntu.
- `yeet vm images` now lists, updates, and prunes official image families independently, including targeted updates such as `yeet vm images update vm://nixos/26.05`.
- VM image manifests can declare the guest default user, system init path, and metadata driver so custom images can use the right guest setup model.

### v0.6.5

- `yeet ssh <vm>` now chooses the VM SSH path from the VM network model: `svc` and `svc,lan` VMs proxy through catch, while LAN-only VMs connect directly to their guest LAN IP.
- Added `yeet ssh --force-proxy <vm>` and stderr transport notices so VM SSH makes the selected path clear before opening the session.

### v0.6.4

- Fixed `yeet init` and catch upgrades on hosts whose login shell is not POSIX-compatible, such as fish.

### v0.6.3

- `yeet upgrade --force` can reinstall the selected public release for local yeet and catch hosts, and `--version vX.Y.Z` lets you target a specific release instead of the latest.

### v0.6.2

- `yeet init` now prompts for a Tailscale OAuth client secret, with `--ts-client-secret` for non-interactive catch enrollment as a tagged host.
- Catch stores that OAuth secret for later `--net=ts` services and fails fast with policy guidance when the requested catch tag cannot be assigned.
- The Tailscale guide now gives first-time users a clear setup checklist for `tagOwners`, OAuth owner tags, catch enrollment, and validation.

### v0.6.1

- `yeet upgrade check` now refreshes the latest release for explicit checks, keeps dev-build rows compact, and `yeet upgrade` exits without prompting when nothing can be upgraded.
- Catch now requires a tagged Tailscale identity for RPC, and first-run setup errors link to the Tailscale policy checklist when the catch node or service tags are not allowed by `tagOwners`.
- Tailscale docs now explain interactive catch bootstrap, unattended auth keys, and the separate OAuth setup used for `--net=ts` service tags.

### v0.6.0

- Added `yeet upgrade check` and `yeet upgrade` so release installs can see and apply yeet/catch updates from verified GitHub assets.
- Interactive commands now show a short update notice when a newer public release is available, without making normal commands depend on GitHub.

### v0.5.13

- `yeet init --install-vm-tools` can install VM host packages during Debian/Ubuntu host bootstrap.
- `vm://ubuntu/26.04` now defaults to the v13 image, with root filesystem features normalized for common LTS host tooling.
- Fresh-host VM access is more reliable: `yeet ssh <vm>` uses catch as the stable proxy hop, and catch warns when Tailscale assigns a different MagicDNS name such as `catch-1`.

### v0.5.12

- `yeet init` can bootstrap fresh Docker hosts with `--install-docker` and now configures Docker's containerd snapshotter before catch starts.
- Remote catch installs now surface Tailscale login progress and support `--ts-auth-key` for unattended catch tsnet setup.
- VM host readiness checks no longer prompt to install VM packages on hosts that cannot run VMs because KVM, TUN/TAP, or host architecture support is missing.
