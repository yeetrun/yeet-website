## Cannot connect to host

- Confirm the catch tsnet node is authenticated and visible with
  `yeet list-hosts`.
- Override host: `CATCH_HOST=<host> yeet status`.
- See [Tailscale](/docs/concepts/tailscale) for tailnet setup details.
- For a step-by-step fresh-host check, run
  [First-Run Validation](/docs/getting-started/first-run-validation).

## `yeet init` needs Tailscale login

Catch uses an embedded tsnet node for RPC. If the node is not already
authenticated, `yeet init` prints a Tailscale login URL while the remote install
runs. Open that URL, approve the node, then rerun `yeet status`.

For unattended setup, create a Tailscale auth key for the catch node and pass it
to init:

```bash
yeet init --install-docker --ts-auth-key=<key> root@<host>
```

If your tailnet ACL only allows tagged catch nodes, make sure the new node is
tagged appropriately after enrollment or use an auth key that applies the
required tags.

## Architecture mismatch

- `yeet run` will fail if the binary arch does not match the host.
- Build with `GOOS=linux GOARCH=<arch>`.

## Docker errors

- Docker compose services require Docker on the host.
- On Debian/Ubuntu-style hosts, `yeet init --install-docker root@<host>` can
  install Docker during bootstrap.
- `yeet init` configures Docker's containerd snapshotter during catch install
  so pushed local images are visible to the host Docker daemon. For manual catch
  installs, confirm `/etc/docker/daemon.json` contains:

```json
{
  "features": {
    "containerd-snapshotter": true
  }
}
```

- If a netns-backed compose service becomes unreachable after networking
  changes, catch will now reconcile that service at startup by recreating the
  compose project when it detects stale docker endpoint wiring for the current
  service netns. Check the catch logs for a message saying it recreated the
  service during netns reconciliation.
- For published compose ports inside `svc`, yeet now owns the service-netns
  DNAT chains directly. If you suspect stale port forwarding, inspect the
  service netns and expect to see `YEET_PREROUTING` and `YEET_OUTPUT` as the
  active chains rather than old direct `OUTPUT ... DNAT` rules.
- If you need to force the same repair manually, restart the service through
  yeet or restart the service's `yeet-<svc>-ns.service` unit and let catch
  reconcile it on the next start.

## Published port changes are rejected

If `yeet service set <svc> -p 443:443` says the change would remove existing
mappings, yeet is protecting the current complete publish list. Include every
mapping you want to keep:

```bash
yeet service set web -p 80:80 -p 443:443
```

To replace the list or clear all published ports, acknowledge that with
`--publish-reset`:

```bash
yeet service set web --publish-reset -p 443:443
yeet service set web --publish-reset
```

Use `yeet info web` to see the live published ports before changing them.

## `svc` network firewall issues

- New Debian and Ubuntu hosts should prefer native `nft`.
- `iptables-nft` is the compatibility path.
- `iptables-legacy` should only appear as a fallback compatibility mode.
- Check that the yeet-owned firewall objects exist:

```bash
sudo nft list table ip yeet
sudo iptables-nft -S YEET_FORWARD
sudo iptables-nft -t nat -S YEET_POSTROUTING
```

- On legacy hosts, replace `iptables-nft` with `iptables-legacy`.
- Verify forwarding is enabled on the host:

```bash
sysctl net.ipv4.ip_forward
```

- If the yeet namespace service needs to rebuild networking state, restart it:

```bash
sudo systemctl restart yeet-ns.service
```

- `catch` now runs host-level netns reconciliation in the background, so
  `yeet status` should return promptly after `catch` restarts. If `catch`
  shutdown or restart appears hung, inspect `journalctl -u catch` for the
  underlying docker or netns command that failed to exit.

## VM payloads do not start

VMs require a Linux catch host with KVM available at `/dev/kvm` and TUN/TAP at
`/dev/net/tun`. Many VPS providers do not expose nested virtualization, so
`yeet init` may warn that VM payloads are unavailable even though containers
work.

Check host capability:

```bash
test -e /dev/kvm && echo kvm=yes || echo kvm=no
test -e /dev/net/tun && echo tun=yes || echo tun=no
```

If KVM is missing, use container, binary, or cron payloads on that host and run
VM payloads on a KVM-capable machine.

## `edit` fails without TTY

- `yeet edit` needs a TTY to launch `$EDITOR`.
- Run from a real shell, not a non-interactive session.

## Clearing staged changes

- Use `stage show` to inspect, then `stage clear` to discard staged changes.

## `copy` destination errors

- One side must be a remote `svc:` or `svc:path`.
- Remote paths are relative to the service data dir (`data/`).
- `yeet copy` defaults to rsync-like `-avz` behavior (archive/recursive,
  verbose, compress).
- Use a trailing slash on the source to copy directory contents (rsync-style).
- Absolute paths are rejected.

## Status output

- Use `--format=json` or `--format=json-pretty` if the table is not useful.

## Service type mismatch

- A service's type is fixed after first install.
- If you installed as docker compose, you cannot later install a binary with
  the same name.
