Use this page when you know what you want to do and need the command shape. For
payload-specific details, use the linked payload guide.

## Deploy with the web form

Use the guided flow for a first deploy or when you do not want to remember the
flags:

```bash
yeet run --web
yeet run --web <svc>
yeet run --web <svc> ./compose.yml
```

The form saves the same service config to `yeet.toml` that the CLI uses.
Runtime errors appear in the browser and local terminal so you can edit and
retry without restarting the command.

## Deploy a compose stack

```bash
yeet run <svc> ./compose.yml
yeet info <svc>
yeet logs -f <svc>
```

After the first successful deploy, rerun the saved service from the directory
that contains `yeet.toml`:

```bash
yeet run <svc>
```

Use [Containers](/docs/payloads/containers) for Compose, Dockerfiles, images,
Python, and TypeScript payloads.

## Run a public image

```bash
yeet run -p 8080:80 <svc> nginx:alpine
yeet status <svc>
```

Replace the full published-port list later with `service set`:

```bash
yeet service set <svc> -p 80:80 -p 443:443
yeet service set <svc> --publish-reset -p 443:443
```

## Push a local image

Use this when you already built a container image on your workstation:

```bash
yeet docker push <svc> <local-image>:<tag> --run
```

## Build from a Dockerfile

```bash
yeet run <svc> ./Dockerfile
yeet logs -f <svc>
```

## Deploy a binary or script

Build binaries for the catch host architecture before deploying:

```bash
GOOS=linux GOARCH=amd64 go build -o ./bin/<svc> ./cmd/<svc>
yeet run <svc> ./bin/<svc>
```

For scripts:

```bash
yeet run <svc> ./script.sh
yeet run <svc> ./script.sh -- --app-flag value
```

Use [Binaries & Scripts](/docs/payloads/binaries) for environment and file
setup.

## Run a cron job

```bash
yeet cron <svc> ./job.sh "0 9 * * *"
yeet cron <svc> ./job.sh "0 9 * * *" -- --job-arg foo
yeet logs -f <svc>
```

Use [Cron Jobs](/docs/payloads/cron-jobs) for timer behavior and update flow.

## Create a VM

```bash
yeet run <vm> vm://ubuntu/26.04
yeet ssh <vm>
```

VM payloads require a KVM-capable Linux catch host. Use NixOS when you want
that image family:

```bash
yeet run <vm> vm://nixos/26.05
```

Use [VMs](/docs/payloads/vms) for host requirements, image cache commands,
networking, file copy, and recovery.

## Update a service

For binaries, scripts, Dockerfiles, and Compose files, update the local payload
and rerun:

```bash
yeet run <svc> ./compose.yml
yeet run <svc> ./bin/<svc>
```

For Compose image refreshes:

```bash
yeet run --pull <svc> ./compose.yml
yeet docker outdated
yeet docker update <svc>
yeet docker update --outdated
```

Use `--force` when you want to redeploy even if yeet detects no payload or
config change:

```bash
yeet run --force <svc> ./compose.yml
```

## Stage before applying

Use staging when you want to upload and inspect a payload before applying it:

```bash
yeet stage <svc> ./bin/<svc>
yeet stage <svc> show
yeet stage <svc> commit
```

Discard the staged payload:

```bash
yeet stage <svc> clear
```

## Edit env, config, or files

```bash
yeet edit <svc>
yeet env edit <svc>
yeet env copy <svc> ./app.env
yeet env set <svc> PORT=8080 LOG_LEVEL=debug
yeet copy ./config.yml <svc>:config/config.yml
```

For VM guests, copy to the VM endpoint:

```bash
yeet copy ./configs/ <vm>:~/configs/
yeet copy <vm>:"/var/log/*.log" ./logs/
```

## Choose networking

Use host ports when you only need a host-level TCP port:

```bash
yeet run -p 8080:80 <svc> nginx:alpine
```

Use `svc` for the private service network, `ts` for a Tailscale service node,
and `lan` for a LAN address on hosts where macvlan and DHCP make sense:

```bash
yeet run <svc> ./compose.yml --net=svc
yeet run <svc> ./compose.yml --net=svc,ts
yeet run <svc> ./compose.yml --net=lan
```

See [Networking](/docs/concepts/networking) before using `lan` on a new host.

## Move a service root

Set the root at first deploy:

```bash
yeet run <svc> ./compose.yml --service-root=/srv/apps/<svc>
yeet run <svc> ./compose.yml --service-root=tank/apps/<svc> --zfs
```

Move an existing service while it is stopped:

```bash
yeet stop <svc>
yeet service set <svc> --service-root=/srv/apps/<svc> --copy
yeet start <svc>
```

Use `--empty` instead of `--copy` when you want a new empty root. See
[ZFS](/docs/concepts/zfs) for dataset-backed roots and snapshots.

## Recover data

List recovery points:

```bash
yeet snapshots list <svc>
yeet snapshots inspect <svc> <snapshot>
```

Clone first when you want to inspect recovered data safely:

```bash
yeet snapshots clone <svc> <snapshot> <svc>-recover
yeet info <svc>-recover
```

Restore in place when you are ready:

```bash
yeet snapshots restore <svc> <snapshot> --stop --yes
```

See [ZFS](/docs/concepts/zfs) for snapshot defaults, retention, and VM recovery.

## Target a host

Here, `<catch-host>` is the catch tsnet hostname, not the SSH machine host.

```bash
CATCH_HOST=<catch-host> yeet status
yeet --host=<catch-host> status
yeet status@<catch-host>
yeet run <svc>@<catch-host> ./compose.yml
```

Save a default host:

```bash
yeet prefs --host=<catch-host> --save
```

## Remove a service

Uninstall the service but keep its data:

```bash
yeet rm <svc>
```

Remove the service data too:

```bash
yeet rm --clean-data <svc>
```

Read the confirmation prompt before deleting data.
