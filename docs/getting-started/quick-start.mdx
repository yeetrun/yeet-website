title: Quick Start
description: Fastest path to a working yeet + catch install.
---

Use this path for the first successful deploy. It installs yeet locally,
bootstraps catch on one Linux host, then runs a disposable container.

## Before you start

You need:

- A Linux host with systemd and SSH access.
- Tailscale admin access for the first catch enrollment.
- A Tailscale OAuth client secret that can create tagged auth keys for catch.

You do not need Tailscale installed on your workstation. yeet uses an embedded
tsnet client for normal commands.

Skip VMs for the first run unless you already know the host has KVM. You can
validate VM support after the container smoke test.

## 1) Install yeet on your workstation

```bash
curl -fsSL https://yeetrun.com/install.sh | sh
```

To install the nightly build instead:

```bash
curl -fsSL https://yeetrun.com/install.sh | sh -s -- --nightly
```

## 2) Prepare Tailscale access

Do this before running `yeet init`. Catch must join your tailnet as a tagged
device, usually `tag:catch`.

In the Tailscale admin console, create:

- An owner tag for yeet, such as `tag:yeet`.
- A catch tag, usually `tag:catch`, owned by `tag:yeet`.
- ACLs or grants that let your workstation reach `tag:catch`.
- A Tailscale OAuth client with the Auth Keys write scope (`auth_keys`) and
  `tag:yeet` selected.

Keep the `tskey-client-...` OAuth client secret ready. Interactive `yeet init`
will ask for it during first setup.

For the minimal policy snippet, see
[Tailscale Setup](/docs/concepts/tailscale#first-time-host-setup).

## 3) Bootstrap catch on a host

Run this from your workstation:

```bash
yeet init root@<machine-host>
```

`<machine-host>` is the SSH target. If you use a non-root user, yeet runs the
remote install with sudo.

During first setup, `yeet init` prompts for the Tailscale OAuth client secret
unless you pass `--ts-client-secret=<secret>` or `--ts-auth-key=<key>`. Paste
the secret you created in the previous step.

If the tailnet policy is not ready, finish
[Tailscale Setup](/docs/concepts/tailscale#first-time-host-setup) before
running `yeet init`.

If Docker is missing on a Debian/Ubuntu-style host, interactive `yeet init` can
ask to install it.

## 4) Confirm yeet can reach catch

After `yeet init`, commands target the catch hostname, not the SSH machine
hostname. If yeet saved a default host, these should work:

```bash
yeet version
yeet status
```

For multiple hosts, pass the catch hostname explicitly:

```bash
yeet --host=<catch-host> status
```

See [How Commands Work](/docs/cli/cli-overview) if `root@<machine-host>`,
`<catch-host>`, and `<svc>@<host>` are still unclear.

## 5) Deploy a disposable service

Start with a small container:

```bash
yeet run -p 18080:80 hello nginx:alpine
yeet status hello
yeet logs hello
```

Check the published port from the host:

```bash
yeet ssh -- curl -fsS http://127.0.0.1:18080/ >/dev/null
```

## 6) Remove the disposable service

```bash
yeet rm --clean-data hello
```

`--clean-data` deletes the service data too, so read the confirmation prompt
before accepting.

## Next steps

- Run the [First-Run Validation](/docs/getting-started/first-run-validation)
  playbook if this is a new host.
- Open [Workflows](/docs/operations/workflows) for day-to-day deploy and update
  commands.
- Open [Payload Guides](/docs/payloads) to choose containers, binaries, scripts,
  cron jobs, or VMs.
