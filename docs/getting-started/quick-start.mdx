title: Quick Start
description: Get from zero to one disposable service without learning the whole system first.
---

Use this path for the first successful deploy. It installs `yeet` locally,
bootstraps `catch` on one Linux host, creates a service workspace, and runs a
disposable container.

The goal is not to make every decision now. The goal is to prove the control
loop works: workstation -> catch -> service -> logs -> cleanup. Once that loop
works, the rest of the manual becomes a set of knobs instead of a scavenger
hunt.

## Before you start

You need:

- A Linux host with systemd and SSH access.
- Tailscale admin access for the first catch enrollment.
- A Tailscale OAuth client secret that can create a tagged auth key for catch.
- Tailscale installed on your workstation and connected to the same tailnet.

Catch uses embedded tsnet, but the local `yeet` CLI does not. After `yeet init`,
normal commands connect to the catch tsnet hostname from your workstation. In
plain terms: your workstation still needs to reach the tailnet address.

Skip VMs for the first run unless you already know the host has KVM. Containers
are a smaller first test. They prove the basic path without asking your host
network to perform a magic trick.

## 1) Install yeet on your workstation

```bash
curl -fsSL https://yeetrun.com/install.sh | sh
```

Nightly build:

```bash
curl -fsSL https://yeetrun.com/install.sh | sh -s -- --nightly
```

## 2) Prepare Tailscale access

Do this before running `yeet init`. Catch must join your tailnet as a tagged
device, usually `tag:catch`. A user-owned catch node is rejected, because a
server daemon with user-device identity is how access control turns into vibes.

Your tailnet policy must also let your setup user reach catch on TCP port
`41548` with the `yeetrun.com/app/yeet` app permissions `read`, `manage`, and
`ssh`. First setup requires all three because `yeet init` enrolls catch,
checks the install, and may run shell-backed validation.

In the Tailscale admin console, open `Trust credentials` -> `Credential` ->
`OAuth`.

You have two choices. The broad one is easy. The narrow one is better once you
care exactly which tags this credential can mint.

- Simple broad setup: choose `All - Read & Write` if you are comfortable giving
  the credential broad Tailscale API access.
- Least-privilege setup: choose custom scopes with Auth Keys write
  (`auth_keys`) and select `tag:catch`. If you plan to use service `--net=ts`
  later, select an owner tag such as `tag:yeet` that owns `tag:catch` and
  service tags such as `tag:app`.

Keep the `tskey-client-...` OAuth client secret ready. Interactive `yeet init`
asks for it during first setup. Treat it like a credential, not like an example
string you paste into chat logs.

For the minimal policy snippet, see
[Tailscale Setup](/docs/concepts/tailscale#first-time-host-setup). For the
permission model and split-role examples, see
[Tailscale Access Grants](/docs/security/tailscale-access-grants).
For service network choices, use [Networking](/docs/concepts/networking) before
deploying `--net=ts`; most services that need Tailscale should use `svc,ts`.

## 3) Bootstrap catch on a host

Run this from your workstation. The target here is the SSH machine host, not
the catch hostname that will exist after setup:

```bash
yeet init root@<machine-host>
```

`<machine-host>` is the SSH target. If you use a non-root user, yeet runs the
remote install with sudo.

During first setup, `yeet init` prompts for the Tailscale OAuth client secret
unless you pass `--ts-client-secret=<secret>` or `--ts-auth-key=<key>`. Paste
the secret you created in the previous step.

Fresh installs also ask where catch should store host data. The default is
`$HOME/yeet-data` on the catch host, with services under that path. This is the
host-side state. Know where it is before you deploy anything important.

If the tailnet policy is not ready, finish
[Tailscale Setup](/docs/concepts/tailscale#first-time-host-setup) before
running `yeet init`.

If Docker is missing on a Debian/Ubuntu-style host, interactive `yeet init` can
ask to install it. On Debian/Ubuntu hosts that expose KVM and TUN/TAP, init can
also ask before installing missing VM tools and preparing VM LAN bridge support.
If the host is not VM capable, yeet warns and containers, binaries, scripts,
and cron jobs still work.

## 4) Confirm yeet can reach catch

After `yeet init`, commands target the catch hostname, not the SSH machine
hostname. This split is slightly annoying and very useful: SSH installs the
daemon; Tailscale operates it. If yeet saved a default host, these should work:

```bash
yeet version
yeet status
```

For multiple hosts, pass the catch hostname explicitly:

```bash
yeet --host=<catch-host> status
```

See [How Commands Work](/docs/cli/cli-overview) if `root@<machine-host>`,
`<catch-host>`, and `<svc>@<host>` are still unclear.

## 5) Create a service workspace

Do this before your first `yeet run`. A successful deploy writes `yeet.toml` in
the current directory. Put that file somewhere boring and intentional.

```bash
mkdir -p ~/yeet-services
cd ~/yeet-services
```

Use this directory for real homelab services too. It keeps the deploy recipe
near the files it describes, which is less glamorous than a database and much
easier to inspect at 11 p.m.

See [Service Workspace](/docs/getting-started/service-workspace) before
deploying third-party Compose apps, env files, Dockerfiles, scripts, or
binaries.

## 6) Deploy a disposable service

Start with a small container. It is disposable on purpose:

```bash
yeet run -p 18080:80 hello nginx:alpine
yeet status hello
yeet logs hello
```

Check the published port from the host:

```bash
yeet ssh -- curl -fsS http://127.0.0.1:18080/ >/dev/null
```

## 7) Remove the disposable service

```bash
yeet rm --clean hello
```

`--clean` deletes the service data and removes the disposable `yeet.toml`
entry, so read the confirmation prompt before accepting. Cleanup flags are not
where you want muscle memory doing product management.
