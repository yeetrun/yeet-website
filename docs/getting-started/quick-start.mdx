title: Quick Start
description: Fastest path to a working yeet + catch install.
---

This is the TL;DR path to a working install. It assumes you have a Linux host
with systemd and SSH access. You do not need Tailscale installed on either
machine; yeet uses an embedded tsnet client for RPC. See
[Tailscale](/docs/concepts/tailscale) for details.

## Before you start

- A Linux/systemd host you can SSH into from your workstation.
- Tailscale admin access for first-time tailnet setup: catch must join with a
  tag such as `tag:catch`, and the OAuth client secret must be allowed by
  `tagOwners`. Follow
  [Tailscale First-Time Host Setup](/docs/concepts/tailscale#first-time-host-setup)
  once per tailnet.
- Docker does not have to be preinstalled on Debian/Ubuntu-style hosts;
  interactive `yeet init` can ask to install it.
- VM payloads require KVM. Skip VMs until after your first container deploy if
  you are unsure whether the host supports them.

## 1) Install yeet locally

Release install (recommended):

```bash
curl -fsSL https://yeetrun.com/install.sh | sh
```

Nightly build:

```bash
curl -fsSL https://yeetrun.com/install.sh | sh -s -- --nightly
```

## 2) Bootstrap catch on a host

Use SSH to install the catch service on a remote host:

```bash
yeet init root@<machine-host>
```

If Docker is missing on a Debian/Ubuntu-style host, interactive `yeet init`
can ask to install it. If the catch tsnet node needs first login, `yeet init`
prompts for a Tailscale OAuth client secret so it can enroll catch with a tag
such as `tag:catch`.

If you use a non-root user, yeet will invoke sudo on the remote.

After install, `yeet run <svc>@<catch-host>` uses the **catch host**
(Tailscale/tsnet hostname), not the SSH machine host. See
[Tailscale](/docs/concepts/tailscale).

For unattended installs, VM host tooling, or direct auth-key setup, see
[Installation](/docs/getting-started/installation).

## 3) Sanity check

```bash
yeet version
yeet status
```

For a fuller post-install smoke test across containers, host services, cron,
VM capability, and optional LAN/ZFS support, see
[First-Run Validation](/docs/getting-started/first-run-validation).

## 4) Deploy something

Start with a disposable container:

```bash
yeet run -p 18080:80 hello nginx:alpine
yeet status hello
yeet logs hello
```

If the host is reachable from your browser, open `http://<machine-host>:18080`.
Or check from the host itself:

```bash
yeet ssh -- curl -fsS http://127.0.0.1:18080/ >/dev/null
```

## 5) Observe and iterate

```bash
yeet logs -f hello
yeet restart hello
```

Re-run the same `yeet run` command to roll a new generation. For an existing
service, a payload-only redeploy reuses the saved run options from `yeet.toml`
and changes only the payload.

When you are done with the disposable service, remove it:

```bash
yeet rm --clean-data hello
```

Next, pick the payload guide that matches what you want to run:
[Containers](/docs/payloads/containers), [VMs](/docs/payloads/vms),
[Binaries and Scripts](/docs/payloads/binaries), [Cron Jobs](/docs/payloads/cron-jobs),
or [ZFS](/docs/concepts/zfs).
