This is an opinionated homelab setup. The install flow assumes:

- A Linux host with systemd.
- SSH access to that host.
- You can reach the host from your laptop (LAN or tailnet). You do **not** need
  Tailscale installed locally; yeet embeds tsnet for RPC. See
  [Tailscale](/docs/concepts/tailscale).

## Host terminology

- **machine host (SSH)**: the host you connect to for
  `yeet init --install-docker root@<machine-host>`.
- **catch host (Tailscale/tsnet)**: the hostname that `yeet` targets for RPC
  (`CATCH_HOST`, `--host`, `@<host>`). After install, this is the tsnet name
  advertised by catch (default `catch` or `--tsnet-host`). See
  [Tailscale](/docs/concepts/tailscale).

## Install yeet

Release install (recommended):

```bash
curl -fsSL https://yeetrun.com/install.sh | sh
```

Nightly build:

```bash
curl -fsSL https://yeetrun.com/install.sh | sh -s -- --nightly
```

From the repo root (source build):

```bash
go build ./cmd/yeet
```

Or install globally:

```bash
go install ./cmd/yeet
```

## Upgrade yeet and catch

Check the local CLI and the selected catch host:

```bash
yeet upgrade check
```

Upgrade both from verified GitHub release assets:

```bash
yeet upgrade
```

In a project with multiple catch hosts in `yeet.toml`, scan and upgrade all of
them explicitly:

```bash
yeet upgrade check --all
yeet upgrade --all
```

Interactive commands may show a short update notice when a newer public release
is available. The notice is advisory: normal commands do not depend on GitHub
being reachable.

`yeet upgrade check --all` scans hosts from the current project and default
host. It does not scan the entire tailnet.

Source builds are not self-updated as release binaries. If you build yeet from
source, update with your normal source workflow, such as `git pull` followed by
`go build ./cmd/yeet`.

### Agent skill (optional)

If you use an AI coding tool that supports `npx skills`, you can install
optional guidance for `yeet` and `catch` workflows:

```bash
npx skills add yeetrun/skills --skill yeet
```

This adds instructions and troubleshooting help for supported AI coding tools.
It does **not** install the `yeet` CLI binary itself.

## Install catch on a host

`yeet init` installs `catch` on the remote host. When run from the repo it
builds locally and uploads the binary; when run from a released yeet it
downloads the latest stable release from GitHub before installing.

Use `--nightly` to install the nightly build instead of the latest stable
release.

```bash
yeet init --install-docker root@<machine-host>
```

If this host has KVM and should run VM payloads, install the VM host tools in
the same pass:

```bash
yeet init --install-docker --install-vm-tools root@<machine-host>
```

If Docker is already installed, or you only plan to run non-container payloads,
the bare init path is fine:

```bash
yeet init root@<machine-host>
```

If the catch tsnet node needs first login, `yeet init` prints the Tailscale
login URL while the remote install runs. For unattended setup, create a
Tailscale auth key for the catch node and pass it with:

```bash
yeet init --install-docker --install-vm-tools --ts-auth-key=<key> root@<machine-host>
```

Or use the mise task:

```bash
mise run init-host -- root@<machine-host>
```

Notes:

- If you pass a non-root user, yeet will run the install with sudo on the
  remote.
- The catch service runs as root.
- `yeet init` configures Docker's containerd snapshotter during catch install
  so pushed images are visible to the host Docker daemon.
- `--install-vm-tools` installs VM filesystem tooling on Debian/Ubuntu hosts
  that expose KVM and TUN/TAP.
- If this is your first host, prefer `--install-docker`; the bare
  `yeet init root@<machine-host>` path is best when Docker is already
  installed or you only plan to run non-container payloads.

After bootstrap, use [First-Run Validation](/docs/getting-started/first-run-validation)
to prove the host can run the payload types you care about.

## Docker requirements (internal registry)

Yeet’s internal registry expects Docker to use the containerd snapshotter so
pushed images show up locally (no pull needed). `yeet init` configures this
automatically during catch install. For manual catch installs or
troubleshooting, ensure the host has this in `/etc/docker/daemon.json`:

```json
{
  "features": {
    "containerd-snapshotter": true
  }
}
```

Then restart Docker:

```bash
systemctl restart docker
```

## VM host requirements

VM payloads (`vm://ubuntu/26.04`) require a Linux catch host with:

- x86_64/amd64 architecture.
- KVM available at `/dev/kvm`.
- TUN/TAP available at `/dev/net/tun`.
- Required tools: `qemu-img`, `zstd`, `e2fsck`, `resize2fs`, `mount`, `umount`,
  and `ip`.

During `yeet init`, catch checks those requirements and prints warnings for
missing host capabilities. On Debian/Ubuntu hosts with the required VM devices
available, pass `--install-vm-tools` to install the missing VM packages:
`qemu-utils`, `zstd`, `e2fsprogs`, `util-linux`, `iproute2`, and
`ncurses-bin`.

ZFS is optional. Install and configure ZFS only if you want dataset-backed
service roots, ZFS snapshots, or VMs with `--zfs`; raw VM disks work without
it. If ZFS is present, `udevadm` is required for ZFS-backed VM disk setup. See
[ZFS](/docs/concepts/zfs) for how yeet uses datasets and snapshots.

## Development toolchain (optional)

If you plan to work on yeet from source, install the toolchain via mise:

```bash
curl https://mise.run | sh
echo 'eval "$(mise activate zsh)"' >> ~/.zshrc
```

Swap `zsh` for your shell (bash/fish/etc.) if needed.

From the repo root, install the pinned tools from `.mise.toml`:

```bash
mise install
```

## Manual catch install

If you want to install without yeet:

1) Build `catch` for the target architecture.
2) Copy it to the host.
3) Run `./catch install` on the host.

Example (from the repo root):

```bash
GOOS=linux GOARCH=amd64 go build -o catch ./cmd/catch
scp catch root@<machine-host>:
ssh root@<machine-host> ./catch install
```

## Catch flags and environment

See the [Catch CLI](/docs/cli/catch-cli) for detailed flags. Common settings:

- `--data-dir` to control where state is stored on the host.
- `--tsnet-host` for RPC connectivity.

## Client defaults

`yeet` stores prefs in `~/.yeet/prefs.json`. You can override at runtime with:

```bash
CATCH_HOST=<host> yeet status
```
