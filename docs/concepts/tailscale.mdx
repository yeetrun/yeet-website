title: Tailscale
description: How yeet uses Tailscale for RPC and per-service networking.

Yeet integrates deeply with Tailscale: catch exposes RPC over an embedded
tsnet instance, and services can optionally get their own Tailscale interfaces
and IPs.

## What yeet uses Tailscale for

- **RPC to catch**: `yeet` talks to `catch` over Tailscale/tsnet. You do not
  need Tailscale installed locally for RPC to work.
- **Per-service networking**: `--net=ts` gives a service its own Tailscale
  interface and IP address.

During `yeet init`, catch starts its own embedded tsnet node for RPC. A plain
first run works:

```bash
yeet init root@<machine-host>
```

If the catch node is not already authenticated, `yeet init` prints the
Tailscale login URL and waits for you to finish enrollment.

The catch node must end up with a tag-based Tailscale identity, not a
user-owned identity. A common tag is `tag:catch`. If your tailnet policy does
not already allow the catch node to use that tag, update `tagOwners` in
Tailscale before or during first setup. For unattended host bootstrap, create a
preauthorized auth key that assigns the catch server tag and pass it with
`yeet init --install-docker --ts-auth-key=<key> root@<machine-host>`.

If you use more than one catch cluster, use tags and Tailscale ACLs/grants to
express the boundary. For example, hosts tagged `tag:catch` can be allowed to
talk to other `tag:catch` hosts while hosts tagged `tag:other-catch` are kept in
a separate group. Admin devices can be granted explicit access to either group
in the tailnet policy; they do not need to share the catch tag.

If another tailnet device already owns the requested catch hostname, Tailscale
may assign a suffixed name such as `catch-1`. Yeet prints a warning during
`init`; target that assigned catch host with `--host=catch-1`, set
`CATCH_HOST=catch-1`, or remove the stale/conflicting Tailscale device and rerun
`yeet init`.

If the catch node joins as a user-owned device, catch refuses RPC and asks you
to fix the Tailscale policy/device tag, then rerun `yeet init`. Passing
`--ts-auth-key=<key>` is the unattended way to make that enrollment tagged from
the start.

## Keys and Tags

Yeet uses two different Tailscale credential paths:

- **Catch bootstrap key**: passed to `yeet init --ts-auth-key`. This key
  authenticates the catch host itself for unattended installs and must assign
  at least one catch server tag, such as `tag:catch`. It is optional for
  interactive first-run setup.
- **Service key source**: used only for services deployed with `--net=ts`. The
  recommended path is `yeet tailscale --setup`, which stores a Tailscale OAuth
  client secret on catch so catch can mint per-service auth keys.

For `--net=ts --ts-tags=...`, Tailscale decides which tags the OAuth client can
assign. A client can mint keys for an exact matching tag set. If you want one
OAuth client to assign different service tags dynamically, define a tag-owner
hierarchy in the tailnet policy: give the OAuth client an owner tag, then let
that owner tag own the service tags you plan to pass with `--ts-tags`.

There is no yeet-side wildcard that bypasses Tailscale tag ownership. The auth
key or OAuth client must be allowed by Tailscale to assign every tag you ask
for.

### Tailnet Policy Checklist

Yeet does not need to own your whole tailnet policy. It does need the policy to
allow the tags you ask Tailscale to assign:

- `tagOwners` must allow catch bootstrap to assign the catch tag, such as
  `tag:catch`.
- `tagOwners` must allow the catch OAuth client to assign any service tags you
  pass with `--ts-tags`.
- ACLs or grants must allow your admin device or admin tag to reach the catch
  tag on TCP port `41548`.
- ACLs or grants for service tags are yours to design; yeet only asks Tailscale
  to enroll the service with those tags.

For example, a small policy might include:

```jsonc
{
  "tagOwners": {
    "tag:catch": ["autogroup:admin"],
    "tag:yeet": ["autogroup:admin"],
    "tag:app": ["tag:yeet"]
  },
  "grants": [
    { "src": ["autogroup:member"], "dst": ["tag:catch"], "ip": ["tcp:41548"] },
    { "src": ["tag:catch"], "dst": ["tag:catch"], "ip": ["tcp:41548"] }
  ]
}
```

If you use a Tailscale OAuth client for `yeet tailscale --setup`, give that
OAuth client the owner tag (`tag:yeet` in the example) so it can mint keys for
the service tags owned by that tag.

## Get Started (OAuth Setup)

Before using `--net=ts`, configure the catch host with a Tailscale OAuth client
secret:

```bash
yeet tailscale --setup
```

This stores a `tskey-client-...` secret on the catch host. When you deploy a
service with `--net=ts`, yeet uses that secret to mint per-service auth keys.
Make sure the OAuth client can assign the service tags you pass with
`--ts-tags`.

If you prefer, you can bypass OAuth and provide your own key:

```bash
yeet run <svc> ./compose.yml --net=ts --ts-auth-key=tskey-...
```

When you pass `--ts-auth-key` directly to `yeet run`, that key is for the
service node, not the catch host. If you also pass `--ts-tags`, provide a key
that already assigns those service tags.

## Service networking (`--net=ts`)

Using `--net=ts` starts a dedicated tailscaled instance for the service and
adds a Tailscale interface inside the service's network namespace. The service
gets its own tailnet IP:

```bash
yeet run <svc> ./compose.yml --net=ts --ts-tags=tag:app
```

Check the IPs:

```bash
yeet ip <svc>
yeet info <svc>
```

`--net` can be combined with other modes (for example, `--net=svc,ts`).

Note: `--net=ts` alone means the Tailscale TAP interface is the only network
device for the service. That can be useful, but it often means no general
internet access unless you route via a Tailscale exit node. A common pattern is
`--net=svc,ts` so the service has a local service IP for outbound access while
still getting its Tailscale address for tailnet traffic.

Tailscale options:

- `--ts-ver=<ver>`: Tailscale version to install for the service.
- `--ts-tags=<tag>`: tags (repeatable or comma-separated).
- `--ts-exit=<node>`: exit node.
- `--ts-auth-key=<key>`: auth key for the service netns.

## Running Tailscale commands for a service

Use `yeet tailscale` (alias: `yeet ts`) to run Tailscale commands inside the
service's netns:

```bash
yeet tailscale <svc> -- status
yeet ts <svc> -- debug daemon-logs
```

`yeet ts <svc> update` runs a **yeet-managed** updater (not Tailscale's official
self-update path). It resolves the latest version on the service's current
track (stable/unstable), replaces the managed `tailscaled` binary, and restarts
the service's tailscaled unit. It prompts for confirmation before applying the
upgrade.

You can pin a specific version:

```bash
yeet ts <svc> update 1.95.112
# or
yeet ts <svc> update --version=1.95.112
```

If you intentionally want the official upstream subcommand behavior, pass it
through explicitly:

```bash
yeet ts <svc> -- update
```

### Tailscale Serve (HTTP proxy + TLS)

Serve is useful for exposing a service over HTTPS on your tailnet. For example,
to proxy a service running on port 3000:

```bash
yeet ts mysvc serve --bg 3000
```

- `serve` can reverse-proxy a local HTTP server (by port or URL).
- HTTPS uses an automatically provisioned TLS certificate.
- Serve requires HTTPS to be enabled on the tailnet; the CLI will prompt you to
  enable it if needed.
- With `--bg`, Serve runs persistently and resumes after reboot; without it,
  you must restart Serve after a reboot.

## Host discovery

`yeet list-hosts` uses your local Tailscale client to discover tagged nodes,
so it requires Tailscale running locally.

## Cleanup

When you remove a service, yeet attempts to delete the corresponding Tailscale
device from your tailnet.
