---
title: Data Layout
description: Where catch stores service state and data.
---

Catch stores live state on each host under its data directory. You usually do
not need to edit these files directly, but the layout helps when you inspect a
host or plan service roots.

## Data directory

Use `catch --data-dir` to set the catch data directory. When yeet installs
catch, it starts the service with an explicit host data directory.
Manual catch runs default to a `data` directory under the current working
directory.

```text
<data-dir>/
  db.json
  services/
  mounts/
  registry/
  tsnet/
```

Use [catch reference](/docs/cli/catch-cli) for catch flags.

## Service roots

By default, services live under:

```text
<data-dir>/services/<svc>/
```

Set a custom service root at first deploy:

```bash
yeet run <svc> ./compose.yml --service-root=/srv/apps/<svc>
```

Use a ZFS dataset as the service root:

```bash
yeet run <svc> ./compose.yml --service-root=tank/apps/<svc> --zfs
```

Inside a service root, the common directories are:

```text
<service-root>/
  bin/
  run/
  data/
  env/
```

- `bin/` contains install artifacts such as binaries, compose files, and units.
- `run/` contains runtime files.
- `data/` contains persistent application data.
- `env/` contains environment files managed by `yeet env`.

## Moving roots

`yeet run` can set the root for a new service. Stop an existing service before
moving its root with `yeet service set`:

```bash
yeet stop <svc>
yeet service set <svc> --service-root=/srv/apps/<svc> --copy
yeet start <svc>
```

Use `--empty` instead of `--copy` when you want a new empty root.

For ZFS-backed moves:

```bash
yeet stop <svc>
yeet service set <svc> --service-root=tank/apps/<svc> --zfs --copy
yeet start <svc>
```

Parent directories or parent datasets must already exist. `yeet service set`
leaves the old root in place.

## Local replay config

`yeet.toml` stores the local recipe for future runs. Catch stores the live root
on the host.

If another checkout changes live service settings, sync them back:

```bash
yeet service sync <svc>
yeet service sync --all
```

## ZFS snapshots

ZFS-backed service roots can get yeet-managed recovery points before deploys,
Docker image updates, and service-root moves. Use the ZFS guide for snapshot
defaults, retention, clone-first recovery, and VM disk behavior:

- [ZFS](/docs/concepts/zfs)
- [Workflows: recover data](/docs/operations/workflows#recover-data)
